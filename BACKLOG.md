## Enforce builder output with JSON Schema structured outputs

> *This was generated by AI during triage.*

**Category:** enhancement  
**State:** needs-triage

### Summary

Require the provider to return builder responses through a machine-enforced JSON
Schema instead of relying on generic JSON-object mode plus prose instructions for
structural correctness.

### Current behavior

The builder instructions describe the generated and unsupported response shapes and
the currently supported operation shapes. The provider requests a generic JSON object,
then deterministic Python code checks exact fields, types, operation names,
references, initial state, and proposed-solution execution.

This safely prevents malformed programs from being accepted, but it allows the model
to return structurally invalid JSON objects that consume a repair attempt. It also
duplicates structural descriptions between model instructions and Python validation.

### Desired behavior

The provider request should use schema-enforced structured output for the complete
discriminated builder response: either `generated` or `unsupported`. The schema should
describe the environment-program and action-invocation shapes supported at the time
the ticket is implemented, including operation-specific required fields and rejection
of unexpected fields.

The schema is only the structural gate. Deterministic Python must continue to own
cross-reference validation, map and initial-state invariants, interpretation freezing,
solution replay, environment freezing, and objective truth.

Builder instructions should remain concise and explain authoring intent and operation
semantics. Do not paste a second handwritten copy of the complete JSON Schema into the
prompt. The model still needs to know what operations mean; the schema only describes
which JSON forms are allowed.

### Key interfaces

- Builder provider request — declares and requests the complete structured-output
  schema rather than generic JSON-object mode.
- Build response — remains the `generated | unsupported` result described by the
  authoritative environment-program contract.
- Environment-program operation definitions — structural shapes used by provider
  output enforcement and deterministic validation must not silently disagree.
- Builder instructions — retain semantics, architectural boundaries, unsupported
  behavior, and repair rules without becoming the structural source of truth.

### Acceptance criteria

- [ ] Provider-backed builder calls request a strict schema for both `generated` and
      `unsupported` response variants.
- [ ] The schema requires all documented fields, rejects unexpected fields, enforces
      nonempty interpretation/reason strings, and distinguishes operation-specific
      condition and effect shapes.
- [ ] Offline tests demonstrate that representative valid responses satisfy the schema
      and malformed fields, wrong types, extra fields, and unknown operations do not.
- [ ] Provider-fake tests verify that builder calls send the strict schema rather than
      generic JSON-object mode.
- [ ] Deterministic reference, initial-state, and solution-replay validation remains in
      Python and is still exercised after structurally valid output is decoded.
- [ ] The acting model never receives the builder's proposed solution or repair
      conversation as a consequence of this change.
- [ ] Builder instructions describe operation meanings and authoring constraints but
      do not embed a duplicate handwritten copy of the complete schema.
- [ ] A test or shared-definition boundary prevents provider schema and Python shape
      validation from silently accepting different operation forms.

### Out of scope

- Adding new generic operations or complete mechanics.
- Replacing deterministic reference checks or solution replay with schema validation.
- Letting the provider or model repair map geometry, rules, or runtime state.
- Executing model-generated code.
- Redesigning the acting-policy response contract unless separately approved.

### Scheduling note

Prefer completing this before the operation vocabulary becomes substantially larger,
or explicitly make it part of the ticket that first centralizes the complete operation
definitions. Adding independently handwritten schema branches after every later
operation would increase drift risk.
